Software Engineering
Software engineering The branch of computer science that seeks principles to guide the development of large, complex software systems.
Integrated development environment (IDE) Systems that combine tools for developing software (editors, compilers, debugging tools, and so on) into a single, integrated package.

Modular implementation The degree to which a system's components may be separated and recombined, often with the benefit of flexibility and variety in use.
Unified Modeling Language (UML) A general-purpose, developmental modeling language in the field of software engineering that is intended to provide a standard way to visualize the design of a system.
Use case Another term for interactions.
Use case diagram A depiction of the proposed system as a large rectangle in which interactions between the system and its users are represented as ovals and users of the system are represented as stick figures.
Data relationships
  1. One-to-one
  2. One-to-many
  3. Many-to-many
  1. A relationship that is exemplified by the association between patients and occupied private rooms in that each patient is associated with only one room and each room is associated with only one patient.
  2. A relationship exemplified by the association between physicians and patients in that one physician is associated with many patients and each patient is associated with one (primary) physician.
  3. A relationship that occurs when including consulting physicians in the physician patient relationship.
Design pattern A predeveloped model for solving a recurring problem in software design.
Software quality assurance (SQA) A process of overseeing and enforcing the quality control systems adopted by the organization.
Software testing:
  1. Glass box testing
  2. Black box testing
  3. Beta testing
  4. Alpha testing
  1. When a software tester is aware of the interior structure of the software and uses this knowledge when designing a test.
  2. Tests that do not rely on knowledge of the software’s interior composition.
  3. When a preliminary version of the software is given to a segment of the intended audience with the goal of learning how the software performs in real-life situations before the final version of the product is solidified and released to the market.
  4. Similar to beta testing, only conducted at the developer's site.
Documentation
  1. User documentation
  2. System documentation
  3. Technical documentation
  1. Explaining the features of the software and describe how to use them.
  2. Describing the software’s internal composition so that the software can be maintained later in its life cycle.
  3. Describing how a software system should be installed and serviced (such as adjusting operating parameters, installing updates, and reporting problems back to the software’s developer).

  3. Best defense against software liability claims
  1. A legal agreement between the owner and user of a software product that grants the user certain permissions to use the product without transferring ownership rights to the intellectual property.
  2. Statements in software licenses of the limitations of liability for the developers of the software.
